    28	### Severity Calibration Notes
    29	
    30	- **Critical** = Interruption recovery failure. ADHD developer loses work reference or state.
    31	- **High** = Persistent cognitive tax. Every interaction pays the cost.
    32	- **Medium** = Friction during specific activities (audits, refactoring).
    33	- **Low** = Meta-friction (affects tooling/process, not direct user flow).
    34	
    35	The ID-centric operations row was elevated from High to **Critical** because losing a task ID after interruption has the highest recovery cost in the table: there is no discovery fallback, no fuzzy search, no "show recent" — only exact-match lookup or manual log scanning.
    36	
    37	Evidence search noise was lowered from Medium to **Low** because it affects PM auditors during evidence triage, not developers during normal task operations.

    39	## Trinity Boundaries (PM vs Memory vs Search)
    40	
    41	### Boundary 1: Status Vocabulary (PM ↔ Memory)
    42	
    43	**Violation**: Status mapping spans PM-owned orchestration model (`TaskStatus` enum) and Memory-owned ConPort progress representation (`TODO/IN_PROGRESS/DONE/BLOCKED` strings), with no declared ownership contract.
    44	
    45	**Ownership recommendation**: PM plane should own the canonical `TaskStatus` enum. Memory plane (ConPort) should accept and return PM enum values. Mapping logic belongs in a single adapter at the boundary, not duplicated in each consumer.
    46	
    47	Evidence: `[Direct code] nl_...models.py.txt L13-L21`; `nl_...conport_mcp_client.py.txt L85-L86`.
    48	
    49	### Boundary 2: Conflict Resolution (PM surface, Memory origin)
    50	
    51	**Violation**: Manual conflict evidence originates from Memory adapter/sync artifacts (`sync.py` manual resolution queue), but the friction surface is PM — the user encounters conflict prompts during PM task operations.
    52	
    53	**Ownership recommendation**: Memory plane owns conflict detection and resolution strategy. PM plane owns the user-facing notification and resolution UI. The current "in-memory fallback" straddles both without clear ownership.
    54	
    55	Evidence: `[Scan hit] 30_memory_burden_search.txt L8` (manual resolution log).
    56	
    57	### Boundary 3: Evidence Noise (Search concern)
    58	
    59	**Violation**: Binary-cache matches and broad regex false positives in evidence generation are Search-pipeline concerns (evidence tooling quality), not PM-domain friction.
    60	
    61	**Ownership recommendation**: This row exists in the friction table for audit completeness but should not drive PM plane design changes. Search/tooling pipeline should add `--no-binary` and domain-scoped patterns.
    62	
    63	Evidence: `[Scan hit] 30_memory_burden_search.txt L2,L9-L11`.
