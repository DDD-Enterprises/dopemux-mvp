# ProcessPool Stability Fix - Implementation Plan

## Problem Analysis
- **Root Cause**: KeyError on `op["kind"]` in `_apply_write_ops` function
- **Trigger**: Concurrent execution with workers=4 changes timing/ordering, exposing missing "kind" field
- **Impact**: Phase A aborts early at partition 21 instead of processing all 252 partitions
- **Current Behavior**: Single bad operation kills entire phase (raw_total=21, raw_failed=0)

## Solution Design

### 1. Make "kind" Access Defensive
**File**: `services/repo-truth-extractor/run_extraction_v3.py`
**Function**: `_apply_write_ops` (line ~6178)
**Change**: Replace `op["kind"]` with `op.get("kind")` and add null check

### 2. Improve Future Processing Error Handling
**File**: `services/repo-truth-extractor/run_extraction_v3.py`
**Function**: `execute_step_for_partitions` (line ~7179)
**Change**: Add logging to track which partition is being processed and enhance error diagnostics

### 3. Add Write Operations Validation
**File**: `services/repo-truth-extractor/run_extraction_v3.py`
**Function**: `execute_step_for_partitions` (line ~7259)
**Change**: Validate write_ops before applying them, add defensive defaults

## Implementation Checklist
