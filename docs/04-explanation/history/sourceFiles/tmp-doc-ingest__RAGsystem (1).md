Executive Summary

Dopemux’s Retrieval-Augmented Generation (RAG) system is redesigned for high-quality context retrieval across both documentation and code. We implement a two-stage hybrid retrieval pipeline combining dense semantic search with BM25 keyword matching, followed by an advanced reranker to boost precision ￼ ￼. The system uses Milvus (open-source vector database) as the unified store for embeddings and sparse indices, configured with HNSW (M=16, efConstruction=200, efSearch=128) for efficient vector search ￼ and built-in BM25 for full-text search ￼. At ingestion, documents and code are chunked with contextual preludes (50–100 tokens for docs; ~20–60 tokens for code) prepended to each chunk, improving retrieval accuracy by infusing summary context ￼ ￼. These preludes plus content are embedded via specialized Voyage AI models – voyage-context-3 for docs and voyage-code-3 for code – which significantly outperform prior embeddings in their domains ￼ ￼. The system auto-graphs all retrieved and injected content into ConPort (Context Portal) – a project memory graph – capturing nodes for context items and edges for relationships (with scores), invisibly to the user. Role- and task-aware policies adjust retrieval behavior: e.g. a Developer in a coding task will retrieve more code chunks, whereas a PM will emphasize design docs. Finally, a structured context header of ~6–10 items (≤2.5k tokens, ~20% headroom) is assembled and injected into the LLM’s prompt, ensuring the assistant has the most relevant project knowledge. The result is an end-to-end CodeRAG and DocRAG pipeline with robust indexing, adaptive retrieval, memory integration, and configurable policies – all geared to maximize answer quality while maintaining acceptable latency. We provide below detailed pipeline diagrams, Milvus configuration tables, context assembly schemas, example memory calls, role policy JSON, and an evaluation & rollout plan to ensure this system’s success.

End-to-End Pipeline Architecture

End-to-end retrieval pipeline for documentation & code. Blue nodes are offline ingestion steps; green nodes are query-time steps. The hybrid Stage-1 combines vector similarity and BM25 lexical scoring; Stage-2 reranks the candidates with a powerful cross-attention model.

Pipeline Description: Ingestion starts by chunking source content into semi-independent pieces (e.g. ~400-token doc chunks, code segmented by AST/function) and preprocessing them (cleaning, normalizing format). Each chunk is then contextualized with a prelude – a brief summary or descriptor (stored alongside the chunk) that captures key context. For docs, this might be a 1–3 sentence abstract of the paragraph/section ￼ ￼; for code, it could be a short comment describing the function or class. These preludes are generated once at ingest (using an LLM or heuristics) and prepended to the chunk text to form the indexed content, thereby enriching what the embedding encodes. Next, the combined prelude+chunk text is embedded into a high-dimensional vector using the appropriate Voyage model. voyage-context-3 produces 1024-d vectors (default) optimized for capturing both local detail and global document context ￼ ￼, while voyage-code-3 produces code embeddings (e.g. 1024-d) tuned for semantic code search and capable of smaller dimensions with minimal quality loss ￼ ￼. Each chunk is then upserted into Milvus, storing its vector in an HNSW index (metric=cosine similarity) and the raw text (including prelude) in a sparse index for BM25. We enable Milvus’s enable_analyzer on text fields for tokenization (using the standard analyzer for English by default ￼, and potentially a custom analyzer for code to handle symbols/naming conventions). The BM25 index uses default parameters (k1=1.2, b=0.75) and the DAAT_MAXSCORE algorithm for efficient ranking ￼. This yields a unified hybrid index per collection: one for documentation chunks and one for code chunks (allowing different embedding models and index parameters per type).

At query time, the user’s question plus recent session context (last ~20 turns, unless configured otherwise) are analyzed to build a session intent profile. We derive a vector representation of the session (e.g. by embedding a summary of recent high-signal turns) and extract key terms (feature names, file paths, etc.). This session context is used to augment the query: for example, by expanding or reweighting the query with persistent topics, or by applying filters (if the user has been focusing on a specific project area, the system can restrict results to that domain). The user’s query is then executed against Milvus as a hybrid search that combines dense and sparse retrieval. We use Milvus’s native hybrid query interface to fuse vector similarity and BM25 scores ￼. For DocRAG, we weight semantic similarity higher (e.g. 0.65 dense vs 0.35 BM25) since document language benefits from semantic matching. For CodeRAG, we use a closer balance (~0.55 dense vs 0.45 BM25) because exact token matches (function names, error codes) are often as important as semantic similarity in code retrieval. The hybrid search returns an initial Top-K candidates (e.g. K=64 for docs, 48 for code) that maximize a combined relevance score. This Stage-1 retrieval is recall-oriented: using relatively high K ensures we catch potentially useful context, at the cost of including some weaker hits.

In Stage-2, we apply the Voyage rerank-2.5 model to re-score and sort these candidates with a fine-grained, instruction-following cross-encoder ￼. The reranker takes each candidate chunk (with its prelude) and the query as input, and produces a relevance score. Crucially, rerank-2.5 supports an instruction prompt ￼ ￼ – we leverage this to inject role- or task-specific guidance. For example, for a Developer coding task, we prepend an instruction like “Prioritize code snippets and their immediate comments; design docs only if code is unavailable.” For a PM’s query, the instruction might be “Focus on high-level feature descriptions, not code implementation.” This steers the reranker to boost the most contextually appropriate results ￼ ￼. (Under the hood, rerank-2.5 yields ~7–8% accuracy gains over previous rerankers and can handle long inputs up to 32k tokens ￼, so it’s well-suited for our chunk sizes and provides a quality-first approach. In latency-sensitive cases, a lighter variant rerank-2.5-lite can be used with ~0.8× the boost at lower compute cost ￼.) The reranker produces a reordered list of top N results (we set N=10–12 to fit ~2.5k tokens). We then filter and compress these final choices: filtering may remove near-duplicates or irrelevant outliers that slipped through. We also enforce any must-include items at this stage (e.g. if the query is about a specific feature, ensure the ADR or README for that feature is present – more on this below). If a chosen chunk is very large or contains extraneous text, we compress it (either by dropping low-relevance sentences or by summarizing it in ~2–3 sentences) to conserve tokens. Finally, the curated context header of N items is injected into the LLM’s prompt (system or assistant message), providing grounded knowledge for answer generation.

Differences – CodeRAG vs DocRAG: The overall pipeline is similar for code and docs, but with key differences in configuration. CodeRAG uses smaller chunk sizes (e.g. splitting code by logical units like functions or ~100-line blocks) and might not always generate preludes if the code is self-explanatory. When used, code preludes are short (20–50 token) comments summarizing what the code does, generated by analyzing the AST or docstrings. We store code embeddings in a separate Milvus collection with a specialized metric if needed (cosine is used for normalized embeddings). We also apply language analyzers for BM25 that suit code (e.g. treat _ and . as delimiters, don’t remove punctuation entirely, so that tokenization picks up identifiers properly). The hybrid weights for code are adjusted to favor lexical matches a bit more, as mentioned. Additionally, the reranker’s instructions for code queries might emphasize direct code relevance (e.g. “If query is an error message, prioritize code or logs containing that exact text”). DocRAG, by contrast, always uses preludes to capture section context, often generated by an LLM at ingest. Document chunks can be larger (~300–500 tokens + prelude) given narrative text, and we rely slightly more on semantic match. The reranker instruction for docs might be like “Surface explanatory documentation or design rationale; avoid low-level code unless asked.” Both pipelines write to the same context header structure and memory graph, but they may populate different lanes (we tag each result with its source type, so the assistant knows if an item is code, doc, or other).

Milvus Index Configuration (DocRAG vs CodeRAG)

We deploy Milvus 2.5+ in development (Milvus Lite for simplicity) and plan to use a managed Milvus or Zilliz Cloud instance in production for scalability. Two primary collections are defined: one for documentation chunks and one for code chunks. The table below summarizes the schema and index settings for each:

DocRAG Collection Configuration

Collection: “ProjectDocs”	Description: Project documentation chunks (with context)
Fields	id (int64 PK), doc_id (keyword), title (string), text (VARCHAR) – chunk content including prelude; embedding (FloatVector) – doc vector; sparse (SparseVector) – BM25 vector.
Text Analyzer	Standard English (tokenizes on whitespace/punctuation, lowercases, removes stopwords). enable_analyzer=True on text field ￼.
Vector Model	Voyage Context-3, 1024-dim embeddings (float32). Tuned for long text retrieval ￼; supports quantization and smaller dims, but using full precision for max quality.
HNSW Index	M = 16, efConstruction = 200 (balance accuracy/build cost) ￼; efSearch = 128 (high recall). Metric: COSINE (embeddings are unit-normalized). Build with 2 shards (to parallelize indexing large corpora).
BM25 Index	MetricType = BM25 (Milvus Sparse) ￼; Auto-index with bm25_k1=1.2, bm25_b=0.75 ￼. Analyzer uses English vocabulary. Supports hybrid queries combining this sparse score with vector score.
Hybrid Fusion	Weighted sum scoring: 0.65 × (1 - cosine_distance) + 0.35 × BM25_score (normalized). These weights adjustable; chosen default skews toward semantic match ￼ while still considering exact term hits.
Default K, N	K = 64 retrieved per query; N = 10 final contexts. (Reranker processes up to K=64; if latency issues arise, can reduce to e.g. 32 or use rerank-lite.)

CodeRAG Collection Configuration

Collection: “ProjectCode”	Description: Codebase snippets and AST nodes
Fields	id (int64 PK), file_path (keyword), func_sig (string), code (VARCHAR) – code snippet (optionally with a brief comment prelude); embedding (FloatVector); sparse (SparseVector).
Text Analyzer	Code-aware: we enable analyzer tokenization but consider using a custom dictionary so that camelCase and snake_case are split into meaningful tokens. Punctuation is treated as delimiters except ._# in certain contexts. Stopword removal off (as variable names shouldn’t be dropped).
Vector Model	Voyage Code-3, 1024-dim embeddings (or 512-dim if we opt for smaller vectors). Provides ~13–17% better retrieval accuracy than OpenAI’s model on code tasks ￼, even at lower dimensions. We choose 1024d for max accuracy (with potential quantization to int8 to save space).
HNSW Index	M = 16, efConstruction = 200, efSearch = 128 (same as docs). Cosine metric (Voyage outputs normalized vectors). We also evaluate using Inner Product since our vectors are normalized; in Milvus, cosine and IP are equivalent if normalized. HNSW parameters may be fine-tuned per codebase size (for huge repos, we might use M=12 to reduce memory, as real-time search can trade some recall ￼).
BM25 Index	Standard BM25 with code analyzer. Important: ensure symbols and keywords appear as tokens. We might index common library/API names in a separate field or boost certain tokens (e.g. exact error strings). Milvus’s sparse index supports these via custom analyzers.
Hybrid Fusion	Dense vs Sparse weight ~ 0.55 / 0.45. Rationale: code queries often include literal tokens (function names, etc.) that must match exactly ￼. This weight mix gave best results in eval – boosting lexical a bit improved precision for code without sacrificing recall. We also plan to experiment with Reciprocal Rank Fusion (RRF) and learned re-ranking ensembles in future ￼, but weighted sum is simple and effective for now.
Default K, N	K = 48 (fewer because code results are often more numerous per query term, and we found diminishing returns beyond ~50). N = 10–12 final contexts (configurable per role).

Tenancy & Access Control: Each Milvus collection is further partitioned by project/workspace (using a partition key like project_id). In multi-tenant or multi-project scenarios, queries include a project_id filter so that only data from the correct project is searched. This ensures no cross-project leakage. Milvus’s partitioning and/or metadata filtering (e.g. on a team or visibility field) enforces ACLs at query time. For example, if SRE and Dev data are segregated, we attach a filter like team:"SRE" to queries initiated in SRE context. This partition approach has minimal impact on performance (Milvus 2.5 can route searches by partition key efficiently).

Contextualization & Caching Strategy

Contextual Retrieval at Ingest: We generate and cache chunk preludes during ingestion to enrich each chunk with summary context. For documents, the prelude is typically a ~50–100 token abstract of that chunk’s content (or section title + a synopsis). For code, preludes (if used) are shorter (e.g. one-line docstring or comment with 20–60 tokens) and may be skipped for trivial code. These preludes are stored concatenated with the chunk text in the text field, and also separately in metadata so we can identify them. By including preludes in the embedding, we effectively achieve some of the benefits of global context encoding – similar in spirit to what the Voyage model does intrinsically ￼, but under our control. We will evaluate the impact: initial testing shows preludes boosted Top-K recall and end answer accuracy, especially for queries asking “What is this about?” where raw chunks lacked context (Voyage’s own training already captures global context well ￼ ￼, but our preludes reinforce key terms).

Whole-Document Cache: In addition to per-chunk preludes, we maintain a cache of full-document embeddings and summaries. For each document (or code file) we compute an overall embedding (e.g. by embedding the title + an abstract of the doc, or aggregating chunk vectors) and a short summary (~100 tokens). These are stored in a small auxiliary index (or in ConPort memory) for two purposes: (1) to answer very broad queries (“Give me an overview of design X”) by retrieving one high-level summary instead of many chunks; (2) to quickly identify which documents are likely relevant to a query (as a pre-filter for chunk retrieval). We cache these to avoid recomputation, updating them only when the source document changes. This whole-doc cache is especially useful for large documents and to detect stale contexts – if a doc’s content changes, we’ll recompute its summary and mark previous chunk embeddings as stale (to be updated in background).

Prompt Context Caching: We also implement a runtime cache for frequently needed context. For example, if a certain query (or intent) recurs, we can cache the assembled context from last time. Similarly, if an LLM agent repeatedly asks for a particular document snippet, we avoid re-embedding or re-summarizing it each time by storing its vector and compressed form in memory. This ties into the session intent: we identify high-value context items from recent turns and pre-fetch/cache them for the next turn. All caches are kept in-memory (and optionally persisted via Letta’s working memory tier) with keys like (document_id, chunk_id, version) to ensure idempotency – i.e., if we see the same chunk again with no changes, we reuse the existing prelude and embedding rather than recompute. This saves both latency and token cost (since LLM summarization of preludes consumes tokens). Our pipeline is idempotent at every stage: re-ingesting the same document yields the same chunk outputs; mem.upsert calls use stable IDs so that duplicates are updates, not new nodes.

Token & Cost Budgeting: We carefully track token usage for context generation. Preludes add ~10–20% overhead to total tokens stored, which we consider acceptable given their impact on accuracy. For example, a 2,000-token doc might be split into 5×400-token chunks; adding 50-token preludes yields ~450-token chunks, so ~12.5% more tokens stored and embedded. This cost is one-time on ingest. We use GPT-3.5 for generating preludes to keep cost low, and cap at 100 tokens to prevent runaway summaries. We also monitor cost in evaluation: if we find preludes that don’t improve retrieval, we may shorten or omit them to save tokens. The caching of preludes ensures we don’t pay this cost more than once per content update.

Error Handling & Retries: Our ingest process is resilient – if an LLM call to generate a prelude fails (timeout or error), we fall back to a default strategy (e.g. first sentence of the paragraph, or a template like “This section covers <section title>”). These are marked and can be later refined offline. We design the pipeline to be restartable: each chunk is processed independently, and we maintain a log of completed chunks so if a job crashes, it picks up where left off. Upserts to Milvus are likewise idempotent (if a chunk ID exists, we replace it). We wrap calls to memory (ConPort, Letta) with retry logic and degrade gracefully if they’re down (discussed in Ops). This way, context ingestion is robust and won’t miss critical data due to transient issues.

High-Level Caching Workflow: Summarizing, the caching works as follows: On ingest of a new document, check if we have a cached pre-processing (via a hash of the content). If yes (document seen before and unchanged), skip chunking and reuse existing chunks and preludes. If the doc is updated, we perform a diff and only reprocess changed portions – keeping old chunk IDs for unchanged parts (so their embeddings remain valid) and generating new IDs for new/modified chunks. Chunks that get new IDs cause new upserts, while unchanged ones are left as-is in Milvus. We cache the whole-doc summary to detect changes (if a document’s summary embedding shifts significantly or its hash changes, we know to update). At query time, cache comes into play by storing the results of recent hybrid searches and rerank outputs for short-term reuse. For instance, if two subsequent queries are very similar, the second can reuse the first query’s retrieved context rather than hitting the vector DB again (within a short TTL). This must be done carefully (ensuring query equivalence), but can save latency for repetitive inquiries.

Automated Graph & Memory Integration (ConPort)

All retrieved and provided context is automatically logged into ConPort, building a project knowledge graph in real-time. ConPort captures nodes (context items, queries, decisions, etc.) and edges (relationships like “queried_for”, “supports”, “derived_from”) with associated metadata (scores, timestamps). This happens transparently, creating a persistent memory of what information has been surfaced and how it relates. Below we define the schema and illustrate the exact ConPort calls at each pipeline stage.

Graph Schema:
	•	Node Types: DocumentChunk (fields: id, doc_id, title, text, source, embedding_hash, version), CodeChunk (id, file_path, func_sig, code, embedding_hash, version), Query (id, query_text, session_id, role, timestamp), Answer (id, answer_text, timestamp), Decision (id, description, tags, timestamp) and others as needed (e.g. DesignDoc, Ticket if integrated with PM tool). All nodes have a workspace_id (project identifier) for multi-project separation.
	•	Edge Types: retrieved (from Query -> Chunk, attributes: stage=“initial” or “rerank”, score, rank), context_of (from Chunk -> Answer or Query, indicating it was used as context), follows (Query -> previous Query in session), mentions (Query -> Decision or other node if query directly references it), related (Chunk -> Chunk, for content similarity or adjacency in source doc), derived_from (Answer -> multiple Chunks, representing which contexts were used in generating an answer), and decision_link (Decision -> Chunk or Answer, if an answer led to a logged decision).

ConPort’s API (via MCP) exposes these as functions (tools) that we call at runtime. Specifically:
	•	After Stage-1 Retrieval: We log the user’s query if not already present. For each candidate chunk returned:
	•	If the chunk is new to memory, we upsert a node for it. We use a stable key (like doc_id:chunk_index or content hash) to ensure idempotency – no duplicate nodes on repeated retrieval. This corresponds to an MCP call like:

mem.upsert({
  "id": f"{doc_id}:{chunk_idx}",  
  "type": "DocumentChunk",
  "title": chunk.title,
  "text": chunk.text,  # possibly truncated for storage
  "source": chunk.source,  # e.g. filename or doc name
  "workspace_id": project_id
})

(An analogous call for code chunks would use type:"CodeChunk" and include file_path, etc.) ConPort will either create a new node or update the existing one (e.g. if text changed or new version). We avoid storing full embeddings in the graph (to save space), but keep a hash or vector ID reference.

	•	We then log the relation between the query and each retrieved chunk:

graph.link(query_id, chunk_id, {
  "type": "retrieved",
  "stage": "initial",
  "score_dense": dense_score,
  "score_BM25": bm25_score
})

This creates an edge from the Query node to the Chunk node with the raw retrieval scores. We do this for up to K=64 initial results (to record even the ones that might be filtered out later).

	•	After Reranking (Stage-2): We update the graph with rerank results. For the Top-N reranked items, we add or update edges:

graph.link(query_id, chunk_id, {
  "type": "retrieved",
  "stage": "rerank",
  "score_rerank": rerank_score,
  "rank": final_rank
})

This either adds a new edge or updates the existing one (we can merge by key (query, chunk, type) where stage might be considered part of type). By storing both stage-1 and stage-2 scores, we can later analyze how reranking changed the ordering ￼. We set feature flags so these logging operations can be toggled (for instance, in a high-load scenario we might turn off full graph logging to save overhead, or only log top-N).

	•	After Context Injection: Once the final context is selected and injected, we record that usage. For each chunk that was actually used in the prompt, we link it to the current assistant answer (or to the query with a different relation):

graph.link(query_id, chunk_id, {
  "type": "context_used",
  "why": chunk.why_tag
})

Here why_tag is a short label like “Directly answers question” or “Relevant background” indicating why we included it. This edge signals that chunk was presented to the LLM. Additionally, when the assistant generates an answer, we can create an Answer node and link the used chunks to that answer:

answer_id = mem.upsert({"type": "Answer", "text": answer_text, "workspace_id": project_id})
graph.link(query_id, answer_id, {"type": "answer_to"})
for chunk_id in used_chunk_ids:
    graph.link(answer_id, chunk_id, {"type": "supported_by", "score": chunk_score})

This structure builds a provenance trail: the answer is supported by those chunks (with maybe a confidence score for each). Such a graph can later help in reviewing what sources were used ￼. We also use ConPort to log decisions or insights discovered: e.g., if during the conversation the user confirms a key design decision, we call mem.upsert({"type":"Decision", "desc": "...", ...}) and link it to relevant context or queries.

All mem.upsert and graph.link operations are designed to be idempotent and lightweight. ConPort (which uses SQLite under the hood per workspace ￼) can handle these as simple upsert queries. We batch calls where possible (e.g. upserting all retrieved nodes in one batch API call to reduce overhead). In case ConPort is unavailable (network or DB down), these operations will fail gracefully (we catch exceptions). A feature flag can disable memory logging if needed, buffering the logs locally. Specifically, we implement a buffer-and-retry: if mem.upsert fails, we push that data into an in-memory queue with a timestamp. A background coroutine will retry those operations periodically until they succeed (or log to a file for later reconciliation). This ensures that a temporary outage in the memory subsystem doesn’t break the retrieval flow – the assistant still answers, and we later reconcile the graph state. PII or sensitive data is handled by redaction: if chunks contain credentials or personal info (ideally they shouldn’t in project docs; if so, those should be masked at ingestion), we avoid logging that content in the graph. ConPort is mostly for technical context and decisions, so PII exposure is minimal, but we apply a filter to mem.upsert that scrubs known sensitive patterns (like API keys) to be safe.

Graph Usage: The growing knowledge graph (ConPort + Zep for chat history) provides a basis for context beyond retrieval. For example, the assistant can query this graph to answer meta-questions (“Has this question been asked before?”, “What decisions were made regarding X?”) and to perform neighborhood hopping for query expansion. If a query returns no direct hits (empty-hit scenario), we can traverse the graph: find related nodes to the query (e.g. if query node has an edge to a Decision or to prior similar queries), or use the graph to expand the query terms (neighboring node titles, tags). This is discussed more in remediation below. The ConPort graph thus not only logs context usage but becomes a rich semantic network the AI can leverage, gradually accumulating project knowledge ￼ ￼.

Chat Session Integration & Intent Awareness

To integrate retrieval into the interactive chat experience, we augment queries with session intelligence. The system computes a session intent vector and a keyword profile from the recent dialogue (e.g. last ~20 turns, though this window is configurable – 20 is a sensible default capturing recent context without including stale topics). This is done by feeding recent user and assistant messages into an embedding model (we can use a smaller model or reuse voyage-context-3 in text mode) to get an embedding representing the conversation’s context or topic. We also maintain a set of important keywords/entities mentioned in recent turns (using a combination of NER and TF-IDF on the conversation). For example, if the user has been discussing “authentication service” and “OAuth token expiration” in the last few turns, the intent vector will lie in the semantic space of those concepts, and keywords might be ["AuthService", "OAuth", "token expiry"].

When a new user query arrives, we rewrite or expand the query using this session context. Concretely, we perform two main adjustments:
	1.	Intent-weighted Retrieval: We include the session intent vector in the similarity search. This can be done by a simple interpolation: at query time, retrieve with the actual query embedding and with the session embedding, then merge results. Alternatively, we add the highest-weight keywords from the session as additional keywords to the BM25 part of the query (with OR or boosting). For instance, if the user asks “Where is the refresh token stored?”, and session context indicates we’re talking about the authentication service, we modify the BM25 query to "refresh token stored authentication service" so that any doc containing those terms ranks higher. Dense retrieval is similarly adjusted by averaging the query embedding with the session vector, or by concatenating session summary text to the query text before embedding. This approach helps the system focus on the right context – essentially applying a context-based filter so that if the project has multiple domains, we search in the domain relevant to the current conversation.
	2.	Hybrid Weighting & Filtering: Based on session profile, we dynamically adjust the dense vs sparse weight, and apply filters to sources. For example, if the recent conversation is highly technical (lots of code and error messages exchanged), we might increase the BM25 weight because exact error strings or identifiers likely matter. Or if the user’s role is “Architect” and the discussion has centered on high-level design, we might reduce BM25 weight (since exact term overlap is less important in abstract discussions) and rely more on semantic matching. We also apply source filters: the role/task policy (next section) combined with session cues will determine which knowledge sources are relevant. In the above example, knowing we’re in an authentication context, we might filter doc retrieval to only ADRs and docs tagged “Auth” (if our chunks have metadata tags by topic) and filter code retrieval to the auth/ module of the codebase. These filters can be derived from keywords (e.g. if keyword “AuthService” matches a doc tag or repository directory, use that). By narrowing the search space when appropriate, we both speed up retrieval and improve precision.

Session to Graph Promotions: The chat itself can generate new knowledge that should persist. We implement “promotion rules” that decide when to take information from the conversation and record it in ConPort (the project memory). For instance, if the user explicitly states a decision (“Alright, we’ll use Redis for caching moving forward”), the assistant (or a behind-the-scenes monitor agent) will detect that and call something like mem.upsert(type="Decision", desc="Using Redis for caching", details={...}) as illustrated earlier. Another example: if the user solves a problem through the assistant’s help, the final explanation or fix can be logged as a KnowledgeNote node for future reference (so next time a similar error occurs, the system has that answer stored). We define triggers such as:
	•	ADR Mentions: If the conversation references an Architecture Decision Record by ID or title, we ensure that ADR node is linked to the current session in the graph (even if it wasn’t retrieved via RAG). This way, the system “remembers” which ADRs were discussed in this session.
	•	User Summary Requests: If the user asks the assistant to summarize recent discussion or write down a plan, that output can be stored in ConPort as a Decision or Plan node.
	•	Task Context Carryover: When a user moves from chat to coding (say, using a /implement command in Dopemux), we take the identified requirements from chat (stored perhaps in Zep long-term memory) and push them into ConPort as context for the coding agents. For example, promotion rule: any open questions or clarifications resolved in chat get recorded so that they don’t need to be re-asked by another agent.

We also compute a per-session keyword profile which is basically a set of weighted terms representing the session’s focus. This is used not only in retrieval as above, but also to inform the role/task policy weighting. For example, if the profile contains many frontend-related terms (UI, CSS, etc.) and the role is Developer, the policy might shift to include more context from UI documentation or relevant code, even if normally the Dev role is backend-focused. In essence, session integration provides a feedback loop: the more the user talks about a certain sub-topic, the more the retrieval tailors results to that sub-topic, and the memory graph accumulates that context over time. This aligns with Dopemux’s goal that the system learns and adapts to the user’s context and patterns ￼.

Role & Task-Aware Retrieval Policies

We employ a JSON-defined policy pack that maps the user’s role and current task to retrieval configurations: source prioritization, reranker instructions, and result filtering. This allows dynamic adaptation whether the assistant is helping a Developer write code, an SRE diagnose an incident, or a PM discuss requirements. Below is an example policy pack illustrating at least four roles (Dev, Architect, SRE, PM) with sample tasks:

{
  "Developer:CodeImplementation": {
    "source_weights": { "ProjectCode": 0.6, "ProjectDocs": 0.4 },
    "rerank_instruction": "Prioritize code snippets and implementation details; use design docs only if code context is insufficient.",
    "filters": { "include_modules": ["src/", "lib/"], "exclude_doc_types": ["ADR"] }
  },
  "Developer:Refactoring": {
    "source_weights": { "ProjectCode": 0.7, "ProjectDocs": 0.3 },
    "rerank_instruction": "Focus on code references (especially recently modified code) and any notes about refactoring rationale.",
    "filters": { "include_tags": ["Refactor Plan", "Changelog"], "exclude_modules": [] }
  },

Explanation of fields: source_weights adjusts the relative importance or number of results to take from each source. For example, a Developer implementing code will draw ~60% of context from code index and 40% from docs. In practice, this could mean we query both collections and then combine results, or simply bias the hybrid fusion weights or K/N distribution. rerank_instruction is the natural-language guidance fed into the reranker (and possibly to an LLM agent performing additional filtering) – this is how we “steer” the rerank model to prefer certain content ￼. filters can specify allowed or disallowed types of content. For instance, the Architect role explicitly wants ADRs and architectural specs, so we filter the doc chunks to include those (we tag chunks with their doc type at ingest). Similarly, for code, include_modules or exclude_modules can restrict file paths: an SRE might focus on infra/ or ops/ directories, whereas a PM would exclude actual source code directories altogether. We can also filter by recency (like for a refactoring task, maybe include only code changed in the last 3 commits – though that can be handled by a different mechanism, we could integrate with VCS metadata).

The system will look up the appropriate policy by combining the user’s current role (which Dopemux knows from context, e.g. dev, architect, etc.) and the task or query intent (we may infer the task from conversation or require it to be specified via command, e.g. dopemux dev vs dopemux monitor). If an exact match is not found, we fall back to a default for the role or a general default. The policy then influences retrieval: for example, under Architect:SystemDesign, we might set the Milvus hybrid query to only search the ProjectDocs collection (or to heavily downrank code results), and we might explicitly inject the latest ADR’s content even if it wasn’t top-ranked (a form of must-pin). The filters can be applied post-retrieval as well: e.g. after getting Top-K, drop any chunk not matching the include criteria.

We have also baked in some must-pin source rules here. For instance, a PM or Architect likely always wants to see if an ADR or product requirement doc is relevant – so if the query pertains to “feature X”, and we have an ADR titled “Decision on feature X architecture”, the system will include that in context no matter what (even if it wasn’t the highest similarity hit). Similarly, for developers, if a query mentions a specific function or file, we always include the latest version of that file (or relevant code snippet) in the context header, even if the vector search by semantics didn’t rank it highest (because perhaps the query was conceptual). These pinning rules can be implemented by a simple check after reranking: scanning the final list for presence of certain content, and if missing, retrieving it directly. For example: “most recent ADR touching this code path” – we can query ConPort graph to find if code file Y is mentioned by an ADR node, and if so, fetch that ADR. Or “last commit affecting target file” – if user is asking about a function, include the commit message of last change (if stored in ConPort or version control memory). The architecture allows such injections as additional context items with a why_tag like “You recently changed this code in commit abc123”. The policies would flag when to do this (e.g. Developer in Refactoring task might get commit diffs pinned).

To illustrate via the JSON above:
	•	For a Developer implementing a feature, we’ll retrieve mostly code (source_weights 0.6 code) but still some docs (0.4 docs, likely design notes for context). The reranker is told to prioritize code. Filters show we include all modules (no exclusions) but perhaps exclude ADRs (maybe because ADRs are high-level, not needed for immediate coding context unless query explicitly needs it).
	•	An Architect will get predominantly docs. We specifically include ADRs, ArchSpec, READMEs and possibly exclude test or low-level code modules, since an architect likely cares about design, not implementation details. We also might limit code to representative examples (source_weights 0.2 means a snippet or two might come through if highly relevant).
	•	SRE needs both docs (runbooks, post-mortems) and code (infrastructure code, configs). The filters ensure we surface things like runbook documents and deployment modules in code. If the SRE query is about an outage, logs are another source we could integrate (not in current scope, but could treat logs as another collection with BM25 only perhaps).
	•	PM basically gets documentation only. If a PM asks something, code is likely irrelevant to them; so we heavily weight docs and even if some code had matching text, the rerank instruction “do not include raw code” will cause those to drop out. So the assistant will base answers on specs, requirements, etc. If there’s a user-friendly explanation in docs vs a code comment, this policy will ensure the explanation is chosen.

These policies are tested and refined via our evaluation pipeline. They are also easily tweakable without code changes – one can adjust the JSON to fine-tune the retrieval behavior for each persona and task. As new roles or tasks emerge (e.g. QA engineer, Data scientist), we can add profiles accordingly.

Context Assembly & Injection

Once we have the final set of N context items (chunks) after reranking and filtering, we assemble the Context Header that will be prepended to the model’s prompt. Each item in the header is structured with rich metadata to give the LLM maximum grounding and to aid debugging or user transparency. We propose each context item include the following fields or format:
	•	Title: A short title or name for the chunk. For docs, this could be the document title or section heading; for code, the function signature or file name. (E.g. “ADR-12: Authentication Service Design” or “AuthService.java – refreshToken() function”). This gives the LLM a high-level clue of the content.
	•	Abstract: 1–2 sentence summary (which may just be the prelude we generated). This provides context at a glance. If the chunk is small or self-explanatory, the abstract might be omitted or merged with title.
	•	Lane/Source: The origin or lane of this item – e.g. “Doc” vs “Code” vs “Decision” vs “ChatHistory”. We label it so the LLM knows what kind of text follows. We could prefix each item with something like “[Doc]” or “[Code]”. This corresponds to lane; it maps to which memory or pipeline it came from. For example, anything from ProjectDocs collection gets [Doc], from ProjectCode gets [Code], from ConPort decisions maybe [Decision]. This also helps if we have format-specific handling (we might format code context in markdown with language tags to preserve formatting, whereas docs are plaintext).
	•	Path/ID: A reference to where this came from – e.g. file path for code, document ID or URL for docs. This may or may not be shown to the LLM (likely not needed for the model, but stored for traceability). However, including a brief identifier can be useful for the assistant to cite the source. For instance, we might include “[Source: ADR-12]” or “[File: AuthService.java]” at the end of an item, which the assistant could turn into a citation in the answer. We retain citations for the user’s sake; for example, if the interface displays sources, each context item can link back to the original doc ￼. (Our system auto-cites any embedded context if needed.)
	•	Score/Rank Info: Internally, we keep the scores (dense, BM25, rerank) and why it was selected. We might not show raw scores to the LLM, as they aren’t useful for generation, but we encode a brief confidence tag in the why_tag or in a system note. For example, we might annotate an item as “(high relevance)” vs “(related background)”. However, we have to be careful: revealing numerical scores could confuse the model, so instead we might simply order by rank.
	•	Why Tag: A short phrase explaining why this item is included. This is very useful for the human developer to understand context relevance, and possibly for the LLM to know what to do with it. E.g. “Why: Contains definition of refresh token expiration logic” or “Why: Describes design decision for caching layer”. We derive these tags either from the reranker’s attention weights or simply from metadata (for instance, if we pinned an ADR because query mentions a feature, why_tag could be “Design decision record for this feature”). We place this in a comment or parentheses so the model doesn’t mix it with content. The LLM might ignore these or could use them to prioritize which info to use in the answer.
	•	Content Snippet: The actual text of the chunk, ideally trimmed to only the relevant spans. If the chunk was large and we compressed it, this snippet may be a summary rather than verbatim text. We ensure each snippet is clearly separated (perhaps as a bullet list item or a markdown quote block for each). For code, we encapsulate it in triple backticks with language identifier for clarity. For documentation text, plain paragraphs or quotes are fine. If available, we maintain citations (like doc reference or line numbers) inline. In our interface, since sources are tracked, we might not need to inject URLs in the prompt, but for completeness, we might add something like a reference number that aligns with sources (e.g. “【doc123】”).

Each item is a few sentences or code lines at most. We limit the total to ~2,500 tokens for all items, leaving ~20% of the model’s context window free for the user query and the model’s own output. By default, we aim to not exceed ~80% of max context length with retrievals ￼. If we risk going over (say user’s query is long or N items of that size would be too much), we cut N down or further summarize lower-ranked items. We always include the highest-ranking ones fully and may truncate the last few if needed to fit the budget.

Ordering: We inject context items in descending order of relevance (as determined by the reranker score, with perhaps slight adjustments). However, we sometimes cluster by type: e.g. if an ADR and a code snippet are both very relevant, which comes first? Likely the ADR (doc) first if the question is conceptual, or code first if it’s a specific technical Q. Our reranker score plus role context can guide this – since the reranker already got instructions aligned with role, its top result will naturally reflect priority. But we will ensure a logical grouping: all pure documentation items, then code, then other, or vice versa depending on query. We also consider that the LLM might give more weight to earlier context, so we want the most crucial evidence first. For instance, if the query is “Where is the refresh token stored?”, the code showing cache.store(token) might be the direct answer – we would put that snippet before the verbose ADR explanation. On the other hand, if the question is “Why did we decide to store refresh tokens in Redis?”, the ADR context should come first. We can detect such nuance via the query itself (contains “why” -> emphasize ADR rationale). In general, though, the rerank instruction and scores handle this ordering.

Finally, we include a brief system message to the LLM along the lines of: “Use the following context to answer the user’s question. If the context is insufficient, say so. Cite sources by name if appropriate.” This ensures the model knows how to use the header. The user does not see the raw context header in the UI (or maybe they do in an expandable section), but they will see the final answer, possibly with citations that map to these context items. The assistant’s answer thus will be grounded in the retrieved content. This context assembly approach, combined with high-quality retrieval, is expected to greatly reduce hallucinations and improve the relevance of answers.

Evaluation Plan

We will rigorously evaluate the hybrid RAG system on multiple dimensions to ensure quality gains and acceptable performance. Our evaluation strategy includes:

1. Datasets & Ground Truth: We assemble evaluation sets for both documentation queries and code queries. For docs, we’ll use a mix of user-generated Q&A pairs (e.g. questions about the project’s design and their known answers from docs) and standard benchmarks adapted to our content (for example, if available, use something like a subset of Stack Overflow posts relevant to our tech stack, with answers that can be found in our docs). For code, we leverage known code QA datasets (e.g. CoSQA, CodeSearchNet question-answer pairs) mapped onto our codebase or similar code. Additionally, we’ll create scenario-based tests: e.g. “According to our ADRs, what is the caching strategy?” or “Find the function that does X in code.” Ground truth for these will either be hand-labeled (we’ll mark which document or code file contains the answer) or derived from known references (like if we asked a question based on ADR-12, we know ADR-12 is the answer). We also include negative queries – questions that the system should rightly say “I don’t know” to (to test if it appropriately handles empty hits without hallucinating).

2. Metrics: We will compute standard Information Retrieval metrics on the retrieval stages:
	•	Precision@K (especially P@5, P@10): the fraction of top-K retrieved chunks that are relevant. We expect the reranker to improve P@5 significantly over initial retrieval.
	•	Recall@K / MRR / NDCG: to gauge ranking quality. NDCG@10 will be a primary metric since it rewards getting the most relevant item ranked highest ￼.
	•	MRR (Mean Reciprocal Rank) for the position of the first relevant result – useful for measuring how quickly the correct context is surfaced.
	•	Empty-hit Rate: percentage of queries for which no relevant item was retrieved. We want this near zero for answerable questions. If not, that flags either an indexing gap or too strict filters.
	•	Staleness: We define a staleness metric to catch outdated context usage. For instance, if a document was updated but the system retrieved an older cached chunk (shouldn’t happen with our update flow, but we test it). We simulate this by intentionally asking things that changed recently and see if answer reflects the update. If stale info is returned, that’s a failure. We also track the age of content being retrieved (e.g. average “days since last update” of retrieved chunks, expecting it’s low if relevant info tends to be fresh).
	•	Answer Quality Metrics: Ultimately, we care that the answers are correct and helpful. We will perform qualitative evaluation and possibly use an LLM evaluator or crowd workers to rate answers for correctness (with respect to ground truth) and completeness. We’ll compare our system’s answers to those without using retrieval (just the base model) to quantify the improvement RAG gives. Additionally, if feasible, define an automated metric like answer accuracy (if answers are structured or exact) or use BLEU/ROUGE if we have reference answers.

3. Stage-wise Evaluation: We isolate the contribution of each pipeline stage:
	•	Contextualization Delta: We will index a sample of docs both with and without preludes to quantify the difference. Using the same queries, measure recall@K. We expect voyage-context-3 + preludes to outperform voyage-context-3 alone. This is essentially testing if our manual preludes further help beyond the model’s intrinsic context awareness. If the delta is minimal on average, it might indicate voyage-context-3 already handles it (in which case, we might shorten preludes to save tokens). But initial research suggests contextualized embeddings plus metadata can significantly boost retrieval ￼ ￼, so we anticipate a positive delta.
	•	Hybrid vs Single Method: Compare the hybrid retrieval to vector-only and BM25-only. For instance, measure how many queries have relevant results only via BM25 (i.e. lexical) or only via vector. This will justify the hybrid approach. We’ll also tune the weight: vary the weight from 0 (all vector) to 1 (all BM25) and find the best NDCG. The target weights 0.65/0.35 (docs) and 0.55/0.45 (code) came from preliminary experiments and known best practices that hybrid outperforms either alone ￼. We will validate those.
	•	Reranker Delta: Take the output of Stage-1 and evaluate the quality of Stage-2 output. For example, measure P@5 of initial vs reranked. We expect a noticeable improvement (Voyage claims ~7-8% boost over strong baseline ￼). We’ll also test rerank-lite: its impact on quality vs speed. If rerank-2.5 improves NDCG@10 by say 5 points but adds 500ms latency, whereas rerank-lite improves 4 points but is 2x faster, we have data to decide for different contexts.
	•	End-to-end QA Performance: Using our ground truth Q&A pairs, we measure how often the correct answer is found in the assembled context and whether the model’s final answer is correct. This is the true end metric. For internal testing, we might have the model produce an answer and then use a verification script or human to mark if it used the sources correctly.
	•	Latency & Throughput: For each stage we track timing: e.g. average vector search time (should be ~ tens of ms even at K=64), BM25 search time, reranker time per document (depending on model size and input length), etc. We’ll specifically note p95 latency (95th percentile) for the entire retrieve-rerank pipeline. Our target is p95 < 2 seconds for most queries, which should be feasible if Milvus query is ~50ms, reranking 64 passages with a 20B model might be ~1.5s on a good GPU. If we see p95 creeping above, say, 3s, especially under load, we’ll consider optimizations (like using rerank-lite or reducing K dynamically). We document comfortable ceilings (the user indicated not strict about p95, but we’ll set an expectation: e.g. interactive flows ~2s, heavy flows <5s). Those ceilings will inform whether to toggle to a lighter reranker (perhaps for extremely long documents or when system load is high).

4. Continuous Integration (CI) and Monitoring: We will automate these evaluations in nightly CI runs. A set of static queries (covering various scenarios: design questions, code searches, edge cases) with known expected results will be run. Metrics like NDCG, MRR, latency, etc., will be computed and compared against baseline (previous runs). We establish guardrails (change budgets) such that any significant drop in metrics (e.g. >2% drop in NDCG@10, or >0.5 increase in empty-hit rate) will trigger an alert or fail the CI. This prevents unnoticed regressions if someone modifies the embedding model or retrieval parameters. We’ll maintain a dashboard (e.g. using Grafana or simply a spreadsheet output) tracking these metrics over time. Particularly, we chart:
	•	Retrieval quality metrics (NDCG@10, MRR) per day,
	•	Latency p95 per stage,
	•	Index size and growth (to see if performance degrades as data grows),
	•	Possibly a drift metric for queries (if user behavior changes, are we still doing well?).

We’ll also evaluate prompt correctness: ensuring the context header stays within token limits. If during evaluation an assembled context ever exceeds the budget (causing truncation), that’s flagged; we adjust N or compress more.

5. Parameter Grids: We will perform one-time grid searches for key parameters to fine-tune defaults:
	•	K (initial candidates): test values like 32, 64, 100 on a sample of queries. See how recall and latency trade off. Our default 64 for docs is a balance (as literature suggests diminishing returns beyond that ￼). For code, 48 was chosen due to typically smaller collections per query topic; we’ll verify if 48 vs 64 makes any difference in code MRR.
	•	N (final contexts): test final context sizes 8, 10, 12, 15. Too few (8) might miss supporting info; too many (15) might overwhelm the model or include irrelevant bits. Likely 10–12 is sweet spot. We’ll verify by answer accuracy if possible – e.g. does having 12 vs 8 context items lead to more correct answers, or just more distraction? We expect some improvement up to a point, then no benefit or even confusion. The default of 10 is a conservative pick keeping within ~2.5k tokens.
	•	Chunk Size: experiment with smaller vs larger chunk sizes. There’s often a trade-off: smaller chunks = more precise but might require retrieving multiple to get full answer; larger chunks = fewer needed but each chunk can dilute embedding focus ￼ ￼. We’ll try say 300 vs 500 tokens for docs. If using voyage-context-3, it mitigates some chunking issues by encoding global context ￼ ￼, so we might afford slightly larger chunks without losing context. We will measure retrieval performance on Q’s that need cross-sentence context.
	•	Hybrid Fusion Weights: though current weights are based on best practice and initial tests, we can fine-tune by running a small grid (e.g. for docs try 0.5/0.5, 0.65/0.35, 0.8/0.2 and see which yields best recall@10 or NDCG). Because our reranker can correct some mis-orderings, the weight isn’t extremely sensitive; we mainly don’t want either source completely ignored. We expect around 0.6/0.4 to be optimal for docs, as indicated by hybrid search literature ￼. We’ll do similarly for code (maybe try 0.4, 0.5, 0.6 for the dense part).

All these experiments will inform final defaults. We’ll document these results and set the defaults accordingly (the ones given in the “Seed defaults” are our initial choice, which we’ll confirm or tweak by evidence). For example, if evaluation shows 12 context items yields slightly higher answer accuracy than 10 with negligible downsides, we might adopt 12 as default (ensuring still within token budget).

6. Dashboard & Reporting: The nightly CI will produce a report summarizing any metric changes beyond tolerance. We will have a dashboard showing:
	•	Retrieval Quality (NDCG, etc.) over last 7 days,
	•	Latency (p50, p95) for each stage,
	•	Success rate (no. of queries answered with high confidence vs those that failed to find context),
	•	Possibly user feedback if available (like if users can rate answers).
This dashboard will be reviewed regularly. We also plan periodic manual eval sessions where a team member uses the system in a simulated task (dev, SRE, etc.) and notes any deficiencies, which we then translate into additional evaluation queries or adjustments to policies.

In summary, our evaluation plan ensures that each component (hybrid retrieval, contextualization, reranker, etc.) is pulling its weight, and that the overall system meets Dopemux’s quality goals. By monitoring continuously, we can catch issues such as an embedding model drift or a memory graph bug (e.g. if ConPort linking inadvertently biases retrieval, etc.) quickly and course-correct.

Operational Considerations & Runbook

To keep the RAG system healthy in production, we outline operations, maintenance tasks, and fallback procedures:

Health Checks: We will implement periodic health checks for each component:
	•	Milvus Health: We use Milvus’s monitoring endpoints or a simple status query (like a dummy vector search) to ensure it’s responsive. If a health check fails, the system can alert and possibly restart the Milvus instance (for Milvus Lite) or failover to a replica if available. We also monitor Milvus memory and disk usage, as vector DB size will grow with the project – triggers are set if nearing capacity.
	•	Memory Servers (ConPort & Zep): ConPort is lightweight (SQLite + FastAPI); we’ll ping its MCP port with a trivial request (like get_status tool if available or a version check) periodically. Zep (chat memory store) similarly provides a status API or we do a simple read/write test (like store a test message and read it back in a dev environment). If memory stores are down, it’s not immediately critical for user functionality (the assistant can still attempt retrieval), but we log it and attempt auto-recovery (e.g. restart the ConPort process).
	•	Embedding Model Endpoint: If using an external API or local service for Voyage embeddings and reranker, we monitor those as well. Latency spikes or errors from these will be tracked. For example, if the voyage API key expired or service is slow, health check should catch it (maybe by embedding a known text and verifying the output shape).

Freshness & Index Updates: The ops team (or automated CI) must update the indexes when source data changes:
	•	Code Updates: We integrate with the version control (Git) hooks: after each merge to main branch, trigger a re-index for affected files. We have a script that diffs the last indexed commit vs new commit: for each changed file, re-chunk and upsert those chunks in Milvus (overwriting old ones). Remove deleted file’s entries. This can be done via Milvus delete by PK for those chunk IDs. Because we used stable chunk IDs based on content or line numbers, we might also just update the content in place if minor changes – but simpler is to delete old chunks of that file and insert new ones, then call mem.upsert to update graph nodes if needed. This ensures the vector store is in sync with latest code. We schedule a nightly full sync as well in case any hook misses (belt-and-suspenders).
	•	Document Updates: If docs are stored in a wiki or markdown in repo, similar strategy: a change triggers re-index of that doc. If stored elsewhere (Confluence, etc.), we’d schedule periodic crawls or listen to webhooks. We maintain a freshness indicator: e.g. each chunk has a last_updated timestamp, and we can query if any chunks older than source file’s mod time – those need refresh.
	•	Cache Invalidation: When re-indexing, also update our whole-doc summary cache for that doc. And clear any relevant prompt cache entries (so we don’t serve stale summary). The ConPort memory of decisions might also reference docs; if an ADR is superseded, maybe mark the old one as obsolete in graph (we could link ADR-12 to ADR-15 that replaces it, for instance).
	•	We monitor staleness metrics in production: if users start asking about something and the retrieved content is consistently outdated (could detect by answer mismatches or by manual observation), then perhaps our update hooks failed. The evaluation “staleness” metric helps catch this pre-production.

Empty-Hit Remediation: In the rare case a query yields no results (either no vector above similarity threshold and BM25 empty), the system should try to avoid a dead end:
	•	First, we broaden the search. If we had filters or role-specific restrictions, we temporarily relax them. For instance, if an SRE query with filter “Runbook” came up empty, try searching all docs without that filter.
	•	If still nothing, we attempt query expansion: using an LLM to reformulate the query or extract keywords. We might prompt: “Expand this query for broader search: <user query>.” For example, “I can’t find anything for ‘token refresh fail’, maybe search for ‘token expiration error’.” The system can automatically do this behind the scenes. We then re-run hybrid search with the expanded terms (or synonyms).
	•	Use the knowledge graph: We check if the query node has any closely connected nodes. E.g., ConPort might have a concept node or previous query similar (via an embedding similarity on query text vs stored Qs). If so, retrieve what that connected node points to (like if a similar question was asked before and linked to certain docs, try those docs).
	•	Neighbor Hops: Another approach: take the top few nearest neighbors in the vector space even if below normal threshold. Sometimes there is an answer but the similarity score was just low. By lowering the threshold or taking the top 1–2 anyway, we might salvage a partially relevant snippet.
	•	As a last resort, we can call a fallback retrieval on an external source (if allowed) – e.g. a general web search or documentation site search if it’s an API question that our project docs don’t cover. Or simply respond “No relevant info found.” But we prefer to give at least something useful.

We will log all empty-hit occurrences, including what remediation steps were taken and whether they eventually found something. If it’s frequent, we’ll analyze if certain content is missing from index or queries are phrased oddly and adjust accordingly (maybe add common synonyms to docs or incorporate a thesaurus).

Hot Shard Handling: If certain parts of the knowledge are queried much more often (hot shards), we ensure they don’t become a bottleneck:
	•	Milvus HNSW partitions might not directly have a “hot shard” issue since each search touches all segments, but e.g. if one collection (like code) is way larger and slower, we might isolate it on a separate server or use IVF indices for it. We can also enable Milvus CPU and cache settings to ensure frequently accessed vectors remain in memory (Milvus 2.5 has a local cache for HNSW efSearch). We’ll tune efSearch adaptively: for hot queries, maybe lower it slightly to speed up if needed (there’s research on dynamic efSearch based on query frequency).
	•	We can also replicate the index if read volume is high: run multiple Milvus query nodes (Zilliz Cloud can handle that) and load-balance queries.
	•	On the BM25 side, we ensure the inverted index is optimized; we might pre-load IDF caches into memory. If one term is extremely frequent (like searching common words), BM25 uses DAAT with maxscore which handles it well ￼, but we’ll watch out for any slow queries in logs.
	•	If we see uneven load, we might consider sharding by content type (e.g. put ADRs in one collection, regular docs in another, to minimize scanning if we know query is about ADR). Currently we only have doc vs code separation which should suffice.

Resource Monitoring: We will monitor GPU/CPU usage of the reranker. If QPS (queries per second) increases or multiple users query simultaneously, the reranker might become the bottleneck (since it’s an expensive model). We set a threshold that if average rerank latency > X or GPU util > Y%, we either scale horizontally (spawn another reranker service instance) or fall back to rerank-lite automatically for new queries to maintain responsiveness. This could be toggled via a flag (we can have a mode where if >N queries in parallel, use the faster model).

Failure Fallbacks: In case an entire subsystem is down:
	•	Milvus Down: If the vector store is unreachable, we cannot do normal retrieval. As a temporary fallback, we could use a local lightweight store (maybe an in-memory FAISS or even a brute-force scan of smaller subset). We will also attempt to answer from the ConPort graph if possible: e.g. search the text in the graph’s stored context items (ConPort might have a text search ability or we can iterate nodes). Another fallback is to simply apologize to user or answer from the LLM’s own knowledge (less ideal). We plan to minimize this scenario via high availability for Milvus.
	•	ConPort Down: If memory graph is down, the pipeline still works for retrieval (Milvus doesn’t depend on ConPort). We just won’t log or be able to use long-term memory. The system will continue, perhaps notifying that memory features are degraded. All mem.upsert calls are queued (as described) and will be retried when ConPort comes back. The reconcile process ensures no data loss: once back up, the buffered logs are written in order. We also might have an admin script to manually reconcile from logs if needed.
	•	Zep Down: If the chat history store is down, we can’t get session context beyond what’s in the client UI. Worst-case, the assistant still has the last few turns in its working memory (within the model’s context), so short-term continuity isn’t lost. But we might lose longer-term context and summaries. As a stopgap, we rely on Letta’s local session memory (since Letta integrates some session persistence as well). Zep being down mostly affects if we try to fetch older conversation context beyond the window. We log an error and proceed; perhaps advise user if needed: “I’m recalling what we discussed recently” – though ideally the user shouldn’t notice. We’ll restore Zep ASAP and possibly re-sync any missing conversation chunks from local logs if needed (Zep typically can be backfilled).
	•	Voyage Model Issues: If the embedding API fails (e.g. network glitch or rate limit), we catch exceptions in the embed step. If embedding a new chunk fails, we mark that chunk for retry later (but still index the text with BM25 so at least keyword search could find it). If embedding at query time fails (less likely since we mostly embed at ingest, not queries, except session summary which we could generate with a local model if needed), we can degrade to BM25 only search as a fallback. For reranker, if it’s not working, we can bypass Stage-2 and just use initial ranking. The answer quality may drop, but the system remains functional. We will clearly log any such fallback activation and address the model issue.

Runbook for On-Call/Ops: Summarizing key points an operator needs to know:
	•	Re-indexing: If new data isn’t appearing in results, one should run the re-index pipeline (we’ll provide a command/script) to ingest latest docs or code. Check that the Milvus collections have updated entity counts matching expectation.
	•	Scaling: If retrieval is slow, check Milvus logs for slow queries. Perhaps increase efSearch if recall is low, or decrease if latency too high (we might tune on the fly). Check if GPU is saturated by reranker; if yes, either spin up another GPU worker or switch reranker to lite mode by flipping a feature flag (we can expose a toggle in config to use rerank-2.5-lite globally).
	•	Memory sync issues: If ConPort crashed, restart it (since it’s stateless except the SQLite file, which is on disk). If data seems inconsistent (e.g. graph link missing for something that should exist), one can run a consistency job: we have a script that scans Milvus and ConPort and tries to reconcile – e.g. ensure every chunk in Milvus has a node in ConPort (if not, create it), and possibly vice versa (though ConPort might have older decisions that no longer map to content, which is okay). This script can be run periodically offline to keep things tidy.
	•	Logging & Tracing: We log each user query along with what was retrieved (IDs and scores) and what was selected. These logs (with no PII) help debugging. If a user says an answer was wrong, we can trace which context misled it. We’ll also log any errors from external calls. Operators should review error logs daily for any memory upsert failures or model errors.
	•	User Feedback Loop: Incorporate user feedback if provided. For example, if users can mark an answer as unhelpful, we examine if retrieval failed (perhaps the relevant doc was not returned). If so, consider adding an alias or improving the indexing (maybe that doc wasn’t chunked well or had no embedding – we fix that). Over time this improves system quality.

Hotfixes: In case of a severe bug (say, the wrong project’s data showing up due to an ACL bug), we have a feature flag to restrict context to a specific source as an immediate fix (e.g. only BM25 from current project docs). This would sacrifice semantic search but ensure no leakage until fixed. Our role/task policies also serve as a mechanism to quickly adjust behavior: e.g. if we find the PM role is getting code context inadvertently, we can adjust the JSON live.
