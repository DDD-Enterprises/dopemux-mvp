# Heuristics Library v1

**Purpose**: Domain-agnostic discovery and extraction patterns for deterministic prompt engineering.
**Scope**: v4 prompt files in Dopemux repo-truth-extractor system.
**Version**: 1.0 (2026-02-23)

## Core Principles

### Determinism First
1. **Stable Keys**: Always prefer path/symbol/name/service_id over generated IDs
2. **Content Hashing**: Use SHA256 for deduplication when no natural key exists
3. **Sort Order**: Normalize arrays by stable keys (alphanumeric, not insertion order)
4. **Unknown Handling**: Emit UNKNOWN for missing data with evidence gap citation

### Evidence Chain
1. **Provenance**: Every fact must cite source (file:line or artifact_id)
2. **Derivation**: Mark computed fields as DERIVED with algorithm reference
3. **Validation**: Required fields must have explicit validation steps
4. **Audit Trail**: Preserve original values when transforming data

## Discovery Heuristics by Phase

### C-Phase (Concurrency & Determinism)
**Focus**: Thread safety, race conditions, atomic operations

1. **Lock Analysis**: Enumerate all synchronization primitives (mutexes, semaphores, RW locks)
2. **Critical Section Mapping**: Identify code regions protected by each lock
3. **Deadlock Detection**: Trace lock acquisition order across call paths
4. **Race Condition Surface**: Flag shared mutable state without synchronization
5. **Atomic Operation Audit**: Verify proper use of atomic types/operations
6. **Thread Pool Analysis**: Map worker threads to task queues and priorities
7. **Memory Barrier Check**: Ensure proper memory visibility guarantees
8. **Context Switch Points**: Identify yield/sleep/block operations

**Failure Modes**:
- Priority inversion scenarios
- Lock convoy and contention hotspots
- False sharing in concurrent data structures
- Thread starvation conditions

### W-Phase (Workflow & Orchestration)
**Focus**: Task decomposition, dependency graphs, state machines

1. **Entry Point Discovery**: Map all workflow triggers (API, event, cron, manual)
2. **State Transition Graph**: Extract states, transitions, and guards
3. **Dependency Analysis**: Build task DAG with edge conditions
4. **Error Path Enumeration**: Trace all exception handling branches
5. **Retry Policy Extraction**: Document backoff strategies and limits
6. **Compensation Logic**: Identify rollback and cleanup handlers
7. **Timeout Analysis**: Map all timeout values and cascading effects
8. **Resource Allocation**: Trace resource acquisition/release lifecycle

**Failure Modes**:
- Orphaned resource leaks in error paths
- Inconsistent state transition guards
- Circular dependency deadlocks
- Unbounded retry storms

### B-Phase (Boundaries & Contracts)
**Focus**: API surfaces, versioning, compatibility

1. **Endpoint Catalog**: Extract all HTTP/gRPC/MCP service endpoints
2. **Schema Evolution**: Trace versioned request/response schemas
3. **Validation Rules**: Document all input validation constraints
4. **Error Code Mapping**: Enumerate all possible error responses
5. **Rate Limit Analysis**: Identify throttling mechanisms and limits
6. **Authentication Flows**: Map authN/authZ decision points
7. **Payload Size Analysis**: Document size limits and chunking
8. **Idempotency Keys**: Identify operations with idempotency guarantees

**Failure Modes**:
- Breaking changes without version bump
- Inconsistent error code usage
- Missing input validation edges
- Authentication bypass paths

## Extraction Procedure Patterns

### Input Processing
1. **Schema Validation**: Validate against declared JSON Schema before processing
2. **Normalization**: Convert to canonical format (trim strings, standardize dates)
3. **Provenance Stamping**: Attach source metadata to every extracted fact
4. **Deduplication**: Remove duplicates using stable content keys

### Evidence Collection
1. **Fact Enumeration**: Extract atomic facts from structured inputs
2. **Relationship Mapping**: Build entity relationship graphs
3. **Temporal Analysis**: Extract timestamps and duration calculations
4. **Spatial Analysis**: Map physical/logical locations and hierarchies

### Transformation Pipeline
1. **Type Conversion**: Safe conversion with fallback to UNKNOWN
2. **Unit Normalization**: Standardize measurements to base units
3. **Derived Computation**: Calculate computed fields with formula citation
4. **Threshold Application**: Apply business rule thresholds

### Output Generation
1. **Schema Validation**: Verify against output schema
2. **Sorting**: Apply stable sort order to arrays
3. **Pagination**: Implement consistent pagination when required
4. **Provenance Preservation**: Maintain evidence chain in output

## Failure Mode Taxonomy

### Data Quality Issues
- **Missing Required Fields**: Field marked required but absent from input
- **Type Mismatch**: Value doesn't match declared schema type
- **Format Violation**: String doesn't match expected format (UUID, email, etc.)
- **Range Violation**: Numeric value outside valid range
- **Enum Violation**: Value not in allowed enumeration

### Processing Errors
- **Transformation Failure**: Computation results in error or invalid state
- **Memory Exhaustion**: Input size exceeds processing capacity
- **Timeout Exceeded**: Processing takes longer than allowed threshold
- **Circular Reference**: Graph contains cycles preventing traversal
- **Stack Overflow**: Recursive processing exceeds depth limit

### Integration Issues
- **Endpoint Unavailable**: External service dependency unavailable
- **Authentication Failure**: Credentials rejected by external service
- **Rate Limit Exceeded**: Throttled by external service
- **Schema Mismatch**: External service returns unexpected format
- **Version Incompatibility**: API version not supported

# PROMPT_S2

## Goal
Produce phase `S2` decision dossier artifacts that compress synthesis results into implementation-ready decision rows. The dossier must preserve evidence traceability, unknown boundaries, and escalation hooks for unresolved conflicts.

## Inputs
- Source scope:
  - `extraction/**/S_synthesis/norm/**`
  - `extraction/**/R_arbitration/norm/**`
  - `extraction/**/T_task_packets/norm/**`
- Required artifacts:
  - `S0_ARCHITECTURE_SYNTHESIS_OPUS.md`
  - `S1_MCP_TO_HOOKS_MIGRATION_PLAN.md`
  - `CONFLICT_LEDGER.md`
  - `RISK_REGISTER_TOP20.md`
- Optional artifacts:
  - `TP_MERGED.json`
  - `FEATURE_INDEX_MERGED.json`
- Constraint:
  - Decision rows must be grounded in supplied evidence only.

## Outputs
- `DECISION_DOSSIER_OPUS.md`
- `S2_DECISION_DOSSIER.md`

## Schema
- Artifact kind: markdown decision tables and registers.
- Canonical writer: `S2` for both outputs.
- Required row fields per decision:
  - `decision_id`
  - `context`
  - `options`
  - `recommendation`
  - `evidence`
  - `risk_ids`
  - `verification_suggestions`
  - `stop_conditions`
- Required sections:
  - Decision table
  - Escalation queue
  - Unknown register
- Output alias requirement:
  - `DECISION_DOSSIER_OPUS.md` and `S2_DECISION_DOSSIER.md` must convey the same decision corpus.

## Extraction Procedure
1. Parse S0 and S1 synthesis outputs into candidate decision statements.
2. Validate each decision against conflict and risk artifacts.
3. Retain only decisions with sufficient evidence; unresolved items enter unknown register.
4. Build escalation queue entries for items requiring PRO conflict, policy, or risk rerank rulings.
5. Render deterministic decision tables and keep row ordering stable.
6. Emit equivalent content to both output files.

## Evidence Rules
- Every decision row must include evidence anchors in canonical format.
- Every recommendation must map to at least one `R8` risk ID.
- Conflicted decisions must cite the associated `R7` conflict entry.
- If evidence is missing, set recommendation to `UNKNOWN` and cite missing artifacts.
- If structured evidence snippets appear, they must include `path`, `line_range`, and `excerpt`.
- No decision may be emitted without traceable evidence.
- Escalation queue items must cite exact anchors that triggered escalation.

## Determinism Rules
- Disallow runtime keys: `generated_at`, `timestamp`, `created_at`, `updated_at`, `run_id`.
- Use fixed headings and table column order.
- Order decisions by `decision_id` or deterministic fallback key.
- Normalize status tokens (`RECOMMEND`, `DEFER`, `UNKNOWN`, `ESCALATE_TO_PRO`).
- Keep both outputs synchronized from the same deterministic representation.
- Avoid free-text drift by using concise, bounded language.

## Anti-Fabrication Rules
- Do not invent decision IDs, risk IDs, or conflict IDs without evidence.
- Do not infer implementation readiness when evidence is incomplete.
- Do not add hidden assumptions to recommendation rationale.
- Do not claim command execution; suggestions only.
- Do not synthesize options that are not represented in source artifacts.
- Do not suppress unknowns to make decisions look complete.

## Failure Modes
- Missing S0 or S1 input artifacts: emit unknown-only dossier with explicit missing inputs.
- Risk linkage absent for a decision: downgrade recommendation to `DEFER` and request evidence.
- Conflict unresolved and no clear winning evidence: enqueue `ESCALATE_TO_PRO`.
- Duplicate decision IDs detected: merge deterministically and retain complete evidence union.
- Alias output mismatch: regenerate both outputs from a single in-memory table model.
- Oversized decision text reducing auditability: enforce concise row summaries and retain detail in evidence anchors.
